# C语言编译过程

写一个 `simple.c`，然后执行：

```bash
gcc simple.c -o simple
```

整个过程大致可以拆成下面几步：

---

## 1. 预处理（Preprocessing）

先做一轮“纯文本替换”：

- `#include` → 头文件直接展开
- `#define` → 宏替换
- `#ifdef` → 条件编译
- 注释会被删掉

为了避免头文件被重复 include，需要用 Header Guard：

```c
#ifndef COMMON_H_
#define COMMON_H_

// 声明

#endif
```

这里要注意：

> 这一阶段完全不懂 C 语义，就是在做字符串替换

输出是 `.i` 文件。

---

## 2. 编译（Compilation）

这一阶段才真正“看懂代码”：

- 语法 / 语义检查
- 生成中间表示（IR）
- 做一些优化
- 最后变成汇编（`.s`）

一个关键点：

> 函数可以没有实现，只要有声明就能过编译

比如：

```asm
call add
```

这里还是符号，不是地址。

---

## 3. 汇编（Assembly）

把 `.s` 转成机器码，生成 `.o`

同时做三件事：

- 指令转二进制
- 建 `symbol table`
- 建 `relocation table`

`relocation table`：

> 对于全局变量的读取 / 存储，以及外部函数调用进行标记，留到链接阶段补全。

---

## 4. 链接（Linking）

链接器生成可执行文件：

- 把多个 `.o` 拼起来
- 找函数实现
- 把之前没填的地址补上

分两种：

### 静态链接

- 从库直接拷需要的目标文件一起链接
- 文件变大，但运行独立

### 动态链接（默认）

- 只保留库的标识
- 运行时再加载动态库
- 节省磁盘空间
- 库更新通常不需要重新编译程序

---

# 第一部分：关键路径（Critical Path）完整流程

同步电路的运行依赖于时钟周期的节奏，数据在寄存器之间传递。

---

## 1. 同步路径概览

```text
        ┌──────────┐
CLK ───▶│ Register A│
        └────┬─────┘
             │
             ▼
      ┌─────────────┐
      │Combinational│
      │   Logic     │
      └────┬────────┘
           │
           ▼
        ┌──────────┐
CLK ───▶│ Register B│
        └──────────┘
```

---

## 2. 时钟上升沿的宏观动作

在每一个时钟上升沿（Rising Edge）触发的瞬间：

- **寄存器 A**：采样并接收当前输入端的新数据 `N`

- **寄存器 B**：采样并接收来自组合逻辑的输出  
  ——这实际上是寄存器 A 在上一个周期接收的数据 `N-1` 经过组合逻辑处理后的结果。

> **核心要求**：寄存器输入必须满足建立时间（Setup Time）与保持时间（Hold Time）约束，否则触发器内部可能进入亚稳态（Metastability），导致输出延迟或逻辑不确定。

---

## 3. 详细流程演进

以下以寄存器 A 的输出为起点进行分析。

### 1. 时钟到 Q 延迟（t_pcq）

时钟上升沿到来后：
- 主锁存器关闭，内部捕获的数据已经稳定（前提是满足 t_setup / t_hold）
- 从锁存器随即打开，主锁存器的状态开始驱动从锁存器内部节点
- 若数据值与从锁存器原先保持的值不同，从锁存器发生翻转
- 翻转后的状态经过一段路径最终稳定的呈现在 Q 端

这一系列内部传播过程所需要的时间称为 **t_pcq（Clock-to-Q delay）**

---

### 2. 组合逻辑处理（t_comb）

信号离开寄存器 A 后，进入组合逻辑网络。

组合逻辑的延迟由：

- 逻辑门延迟
- 导线传播延迟
- 扇出负载
- 晶体管驱动能力

等共同决定。

信号经过逻辑运算后，到达寄存器 B 输入端所需的时间称为：

```text
t_comb
```

这通常是整条同步路径中最大的延迟来源。

---

### 3. 门前数据建立（t_setup）

处理后的信号到达寄存器 B 输入端后，必须在下一个采样边沿到来之前提前稳定。

这一段必须提前稳定的时间称为：

```text
t_setup（建立时间）
```

建立时间存在的根本原因是：

```text
触发器内部需要足够时间，
使内部状态已经明显偏向某个稳定逻辑态
```

这样当采样边沿到来、内部正反馈结构开始主导状态时，系统才能可靠地收敛到确定逻辑值。

如果输入变化过晚，则在采样瞬间，内部节点可能仍接近中间平衡状态，从而导致亚稳态（Metastability）。

---

## 4. 关键路径延迟公式

为了保证系统正常工作，一个时钟周期 `T` 必须能够覆盖整条数据路径的传播时间：

```text
T ≥ t_pcq + t_comb + t_setup
```

如果时钟周期小于该延迟，则在下一个时钟边沿到来时：

- 新数据可能尚未完成组合逻辑传播
- 或寄存器内部尚未建立稳定状态
- 导致建立时间违例（Setup Violation）

此时触发器可能输出错误结果，甚至进入亚稳态。

---

# 主锁存器与从锁存器

为什么寄存器需要建立时间（Setup Time）与保持时间（Hold Time）？

这需要深入到主从 D 触发器（Master-Slave D Flip-Flop）的内部。

---

## 1. 为什么需要建立期（t_setup）？

寄存器的核心是由主锁存器（Master Latch）和从锁存器（Slave Latch）串联组成的。

### 主锁存器透明阶段

当主锁存器处于透明状态时（具体对应哪个时钟电平取决于实现方式），外部输入可以传播到主锁存器内部节点。

### 建立倾向（t_setup 窗口）

在采样边沿到来之前的一小段关键时间内，外部输入必须保持稳定。

这并不只是“给电容充电”，更重要的是：

```text
让锁存器内部节点已经明显偏向某个稳定状态
```

例如内部节点已经形成：

```text
A≈0V
B≈1V
```

或：

```text
A≈1V
B≈0V
```

这样当内部正反馈闭合时，系统才能快速、可靠地收敛到确定逻辑状态。

如果输入在采样边沿前变化过晚，内部节点可能仍接近中间状态（如两侧都接近 `0.5V`），从而导致亚稳态（Metastability）。

---

## 2. 为什么需要保持期（t_hold）？

### 锁存切换瞬间（采样边沿来到）

当采样边沿来到时，主锁存器逐渐关闭，同时内部反馈电路（由交叉耦合反相器组成）开始主导内部状态。

### 再生放大

如果内部节点已经具有明确倾向，正反馈会迅速完成再生过程，将状态收敛到稳定逻辑电平。

例如：

```text
0.1V → 0V
0.9V → VDD
```

### 防止污染（t_hold）

由于真实电路中的传输门与时钟信号都存在有限延迟，锁存器不会在采样边沿瞬间完全隔离输入。

因此，在采样边沿之后的一小段时间内：

```text
输入仍必须保持稳定
```

否则新的输入变化可能继续传播进主锁存器，扰动尚未完全稳定的内部状态，从而导致保持时间违例（Hold Violation）。

---

## 3. 数据交接与输出

与此同时，主锁存器与从锁存器的透明状态发生切换：

1. **从锁存器动作**：从锁存器进入透明状态，允许主锁存器中的状态向输出方向传播。

2. **物理传递**：如果主锁存器已经稳定到确定逻辑状态，其输出会强力驱动从锁存器内部节点，使从锁存器迅速收敛到相同逻辑值。

3. **Q 端稳定**：经过有限传播延迟后，输出端 `Q` 呈现出合法、稳定的新逻辑电平。

---

## 4. 周期循环

当时钟进入下一相位后：

- 主锁存器重新进入透明状态
- 从锁存器关闭并保持当前数据
- 电路准备接收下一轮输入数据

注:可能存在谬误部分仅供参考

---

# 五级流水线数据通路

单周期数据通路的缺点在于一个周期当中的不同组件处于空转状态，并没有充分使用硬件，导致 CPU 速度并不快，吞吐量并不高。由此衍生出了流水线（Pipeline）的概念，通过流水线寄存器将单周期划分为几个阶段，使得同一个周期所有组件都能被不同的指令充分利用。

## 五级流水线架构

流水线分为五个 Stage（阶段），每个阶段由流水线寄存器划分：

- **IF (Instruction Fetch)**：取指并更新程序计数器 (PC)。

- **ID (Instruction Decode)**：译指、读取寄存器文件以及生成立即数。

- **EX (Execute)**：执行阶段，处理算术逻辑运算和分支比较。

- **MEM (Memory)**：读取内存或者写入内存。

- **WB (Write Back)**：写回寄存器文件。

### 流水线寄存器

- **IF/ID**：保存指令以及 PC。

- **ID/EX**：保存读取的寄存器数据、立即数、PC、指令。

- **EX/MEM**：保存 ALU 的输出、rs2 数据、PC、指令。

- **MEM/WB**：保存 ALU 的输出、MEM 的输出、PC+4、指令。

---

# 指令执行流程示例

假设有五条指令 ${I_1, I_2, I_3, I_4, I_5}$（以下都满足建立时间）：

## 1. 第一个时钟上升沿：I₁ 进入流水线

- 来到 IF 阶段，等待时钟到 Q 的时间，PC 输出指令地址进入 IMEM。等待读取延迟后，指令来到流水线寄存器；同时 PC 也会输出到相应的流水线 PC 寄存器，并且输出到加法器加四，经过 MUX 延迟来到下一个 PC 寄存器。

---

## 2. 第二个时钟上升沿：I₂ 进入流水线

- I₂ 来到 IF 阶段并重复 I₁ 在 IF 的流程。

- I₁ 来到 ID 阶段，等待时钟到 Q 的时间，指令被送入寄存器文件开始读取。同时 PC 继续传播，指令传入控制单元、立即数生成器以及对应的指令寄存器。经过一定的延迟生成控制信号（如 ImmSel），随后立即数生成器输出至相应的流水线寄存器（通常寄存器文件的输出会稍晚一点到达流水线寄存器）。

---

## 3. 第三个时钟上升沿：I₃ 进入流水线

- I₃ 来到 IF 阶段，I₂ 来到 ID 阶段。

- I₁ 来到 EX 阶段，同样经过时钟到 Q 的时间，PC、rs2、指令继续传播。指令进入控制单元生成对应的控制信号；rs1 和 rs2 的比较结果传入控制单元，经过延迟生成 PCSel 传入 IF 段的 MUX（这也是后续控制冒险的来源）。同时根据控制信号决定 PC / rs1 / rs2 / imm 谁通过 MUX 进入 ALU 进行运算，经过一定延迟后来到 ALU 流水线寄存器。

---

## 4. 第四个时钟上升沿：I₄ 进入流水线

- I₄ 来到 IF，I₃ 来到 ID，I₂ 来到 EX。

- I₁ 来到 MEM 阶段，经过时钟到 Q 的时间，PC 继续传播，指令继续传播。控制单元决定是读取还是存储内存，经过一定的读取延迟后来到对应的 MEM 流水线寄存器。

---

## 5. 第五个时钟上升沿：I₅ 进入流水线

- I₅ 来到 IF，I₄ 来到 ID，I₃ 来到 EX，I₂ 来到 MEM。

- I₁ 来到 WB 阶段，经过时钟到 Q 的延迟后，将指令送入控制单元以决定 PC+4、ALU 输出、MEM 输出谁通过 MUX，以及是否写入寄存器文件。
在一个周期中，不同的指令依靠这些寄存器同步执行，指令在每个阶段被独立译码生成对应的控制信号。

## 总结

可见一条指令的执行时间变得更长了（需要五个周期，且每个周期都有流水线寄存器的时钟到 Q 延迟等开销），但是**吞吐量变高了**。

在第五个周期开始（即四个周期后），此后的每一个周期结束都能输出一条指令。

由于我们将任务切分，时钟周期大幅缩短，周期的长度取决于耗时最长阶段的**关键路径时间**（通常是 ALU 或 MEM），否则会造成最小周期违例。

---

# 流水线冒险 (Hazards)

流水线在提高频率和吞吐量的同时，也引入了冒险。冒险分为三类：结构冒险、数据冒险、控制冒险。

---

## 1. 结构冒险 (Structural Hazard)

硬件资源抢夺。例如在 IF 阶段和 MEM 阶段，不同的指令需要同时访问内存。

### 解决方式

增加硬件资源。例如采用哈佛架构（分离的指令存储器和数据存储器）来避免内存访问冲突。

---

## 2. 数据冒险 (Data Hazard)

前后指令存在数据依赖关系。例如：

```assembly
add s0, t0, t1
add s0, s0, s0
```

前文流程可知，`s0` 需要在第一条指令的 WB 阶段后才会被写回；而第二条紧随其后的指令，在 ID 阶段读取寄存器文件时，第一条指令才堪堪到达 EX 阶段。此时第二条指令会拿到错误的旧值进行计算。

### 数据冒险的解决思路

#### 1. 停顿 (Stall / Bubble)

引入气泡，使得第二条指令在第一条写回后再执行。

加入气泡指的是：检测到冲突后，将 PC 寄存器的写使能改为 0，IF/ID 流水线寄存器的写使能设为 0（保持当前状态），同时将 ID/EX 寄存器的控制信号清零（相当于插入一条 `addi x0, x0, 0` 即 NOP 指令）。

但这会使得 CPI (Cycles Per Instruction) 变高，性能下降。

---

#### 2. 寄存器半周期读写 (Double Pumping)

支持在 WB 阶段的前半周期写入，ID 阶段的后半周期读取。这样可以少停顿一个周期。

只要：

```text
控制信号延迟 + MUX延迟 + 建立时间 + 写入时间 + 读取时间
```

在一个时钟周期内即可支持。

---

#### 3. 数据转发 (Data Forwarding / Bypassing)

不用老老实实等待写回完毕。

假设第一条指令在 MEM 阶段（第四周期），第二条指令在 EX 阶段，此时只需从 EX/MEM 的 ALU 结果寄存器拉出一条导线，配合检测逻辑和 MUX，直接将结果作为当前 ALU 的输入即可。

如果是第三条指令和第一条指令冲突，则从 WB 阶段将MUX的输出数据拉回 EX 阶段。

### 转发检测逻辑（EX 级与 MEM 级转发）

```c
// EX 级冒险转发 (Forwarding from EX/MEM to EX)
if (EX_MEM.RegWrite && EX_MEM.rd != 0 && EX_MEM.rd == ID_EX.rs1)
    ForwardA = 10;

if (EX_MEM.RegWrite && EX_MEM.rd != 0 && EX_MEM.rd == ID_EX.rs2)
    ForwardB = 10;


// MEM 级冒险转发 (Forwarding from MEM/WB to EX)
// 注意：需要确保不会覆盖更近的 EX 级转发

if (MEM_WB.RegWrite && MEM_WB.rd != 0 &&
    !(EX_MEM.RegWrite && EX_MEM.rd != 0 &&
      EX_MEM.rd == ID_EX.rs1) &&
    MEM_WB.rd == ID_EX.rs1)
{
    ForwardA = 01;
}

if (MEM_WB.RegWrite && MEM_WB.rd != 0 &&
    !(EX_MEM.RegWrite && EX_MEM.rd != 0 &&
      EX_MEM.rd == ID_EX.rs2) &&
    MEM_WB.rd == ID_EX.rs2)
{
    ForwardB = 01;
}
```

---

### Load-Use 冒险（无法通过纯转发解决）

```assembly
lw s0, 0(a1)
add s0, s0, s0
```

对于 `lw` 指令，数据要在 MEM 阶段结束才能从内存读出，因此无法在第一条指令的 EX 阶段结束时把数据转发给第二条指令的 EX。

必须**停顿一周期**，然后再使用转发。

### Load-Use 停顿检测逻辑（在 ID 阶段检测）

```c
if (ID_EX.MemRead &&
   (ID_EX.rd == IF_ID.rs1 ||
    ID_EX.rd == IF_ID.rs2)) {

    // 触发 Stall：
    // 保持 PC 和 IF/ID，
    // 将 ID/EX 清空为气泡
}
```

---

## 3. 控制冒险 (Control Hazard)

分支指令导致的执行流不确定。

```assembly
beq t1, t2, lab
add s0, t0, t1
add s0, s0, s0

lab:
sub s1, s0, s3
```

决定是否跳转是在 EX 阶段（或优化后的 ID 阶段），此时流水线已经取入了后续指令。

如果分支发生跳转，流水线里已经进入的指令就是错误的。

### 控制冒险的解决思路

#### 1. 停顿 (Stall)

遇到分支就停顿，直到算出结果，但太慢。

---

#### 2. 静态分支预测 (Static Branch Prediction)

固定预测分支“不跳转”（总是执行下一条）。

如果不跳转，毫无停顿；如果发生了跳转，则需要**冲刷 (Flush)** 掉流水线上已经取入的错误指令。

具体做法和气泡类似，使用 MUX 将 IF/ID 和 ID/EX 的指令替换为 `nop` (`addi x0, x0, 0`)，但**不需要**锁死 PC 的写使能，因为要跳到新的目标地址。

---

#### 3. 动态分支预测 (Dynamic Branch Prediction)

处理器在运行时根据分支指令的历史执行行为来预测其跳转方向。

通过维护分支历史表等硬件结构记录过往结果，提前取指。这种方法能更准确地适应程序的实际运行模式，显著提升处理器性能。

---

# 计算机体系结构铁律

## 性能铁律 (Iron Law of Performance)

衡量一段程序执行时间的公式：

$$
\text{CPU Time} =
\frac{\text{Instructions}}{\text{Program}}
\times
\frac{\text{Clock Cycles}}{\text{Instruction}}
\times
\frac{\text{Seconds}}{\text{Clock Cycle}}
$$

也可以简写为：

$$
\text{Time} =
\text{Instruction Count}
\times
\text{CPI}
\times
\text{Clock Cycle Time}
$$

- **Instruction Count (IC)**：取决于算法、编程语言、编译器和指令集架构 (ISA)。

- **CPI (Cycles Per Instruction)**：取决于计算机的微架构设计（如流水线结构、缓存设计）。

- **Clock Cycle Time (时钟周期时长)**：取决于硬件实现和半导体工艺制程。

---

## 效能铁律 (Iron Law of Power / Energy)

衡量处理器能耗的核心公式分为能量和功率两个维度。

### 1. 单程序能耗 (Energy per Program)

$$
\text{Energy} =
\frac{\text{Instructions}}{\text{Program}}
\times
\frac{\text{Energy}}{\text{Instruction}}
$$

通过优化指令数或采用低功耗硬件设计来减少每次运算消耗的焦耳。

---

### 2. 动态功耗公式 (Dynamic Power)

$$
\text{Power} =
\text{Capacitance}
\times
\text{Voltage}^2
\times
\text{Frequency}
$$

即：

```text
P = C * V² * f
```

降低电压 ($V$) 是最有效的降功耗手段，同时降低频率 ($f$) 也能减小瞬时发热，但会影响上述性能铁律中的时钟周期。
