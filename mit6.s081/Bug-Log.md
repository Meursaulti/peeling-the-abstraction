**日期：** 2026-06-26

**实验：** MIT 6.S081 Lab 1: Xv6 and Unix utilities (primes)

### 1. 现象观察

- **错误类型：** 文件描述符泄漏 (File Descriptor Leak) 导致的系统资源耗尽。
    
- **预期行为：** 程序利用 CSP（通信顺序进程）思想，通过多进程管道级联正确筛选并打印指定范围内的素数，且所有进程正常退出。
    
- **实际表现：** 在小范围数据（如 $N \le 35$）下输出正常，无报错。当测试范围调大（如范围达到 40+）时，程序会因底层管道创建失败或进程创建失败而异常退出，无法继续输出后续数据。

### 2. 核心复现代码

以下是导致 FD 跨代累积的核心逻辑切片：



``` c
void primes(int *left) {
    if (fork() == 0) {
        close(left[1]); // 关闭了当前层级不需要的写端
        
        // ... 业务读取逻辑 ...

        int right[2];
        pipe(right);
        
        // 触发递归调用与下一次 fork，使后代隐式继承当前层级仍打开的 left[0]
        primes(right); 

        // ... 过滤并写入 right[1] ...
        
        close(left[0]);
        close(right[1]);
        wait((int *)0);
        exit(0);
    }
}
```

### 3. 分析

- 在 xv6 操作系统中，`fork()` 会让子进程完整、全量地复制父进程此刻打开的所有文件描述符（FD）表。
    
- 当子进程创建了 `right` 管道并调用 `primes(right)` 继续递归时，父进程传下来的 `left[0]`（读端）依然处于打开状态。下一代进程诞生后，不仅继承了新的 `right` 管道，还**隐式继承**了祖父辈的 `left[0]`。
    
- 在下一代进程的递归上下文中，传入的形参已经对应新的 `right` 数组，祖父辈遗留的 `left[0]` 不再作为当前层显式管理的对象，导致该 FD 失去被及时 `close()` 的机会。
    
- 每递归一层，子进程体内就会多累积一个无用的祖先读端。当素数筛选层级达到一定深度时，FD 表会被快速填满，导致后续的 `pipe()` 直接返回 -1，程序无法继续创建新的筛选层级。
    

### 4. 验证与发现

1. 为验证程序正确性与梳理思路，脱离代码，在纸上画出管道通信流和 FD 分配表。推演通信状态时，发现子进程状态中残留了属于上一代的不必要 FD。
    
2. 将素数测试范围上限由默认值调大至 40 个以上，观察到程序无法继续正常输出，从而验证了纸面推演的发现。
    

### 5. 解决方案

解决 `fork()` 带来的跨代 FD 继承问题。通过额外传递参数，显式地将“祖先遗留的读取端”透传给子进程，并在子进程初始化时进行定向关闭。

**核心修复代码：**


``` c
// 新增 stale_fd 参数用于接收上一代的残留描述符
void primes(int *left, int stale_fd) {
    if (fork() == 0) {
        close(left[1]);
        close(stale_fd); // 精准释放祖先遗留的、当前不需要的读端

        // ... 业务逻辑 ...
        
        int right[2];
        pipe(right);
        
        // 将当前正在读取的 left[0] 作为下一代的 stale_fd 传下去
        primes(right, left[0]); 
        
        // ...
    }
}

int main(void) {
    // ...
    // 主进程中使用 dup() 复制一个描述符，作为第一个子进程可安全关闭的 stale_fd 占位符
    int stale_fd = dup(pipefd[1]); 
    primes(pipefd, stale_fd);
    
    // 在 main 进程中关闭 dup() 产生的 FD
    close(stale_fd); 
    // ...
}
```
